<objective>
Ensure the `task_type` field in OUTCOMES.md records is consistently populated and that the memory-manager's recall operation exposes task_type groupings for archetype-weighted scoring in Plan 06-03.
</objective>

<context>
- OUTCOMES.md schema already includes `Task Type` column (memory-manager Section 2)
- Current schema: `| ID | Date | Branch | Phase | Plan | Agent | Task Type | Outcome | Importance | Tags | Summary |`
- Memory-manager store operation (Section 3) records outcomes after plan execution
- Memory-manager recall operation (Section 4) computes memory scores per agent: `success_rate * 3.0 + avg_importance * 0.5`, weighted by recency decay
- Current recall does NOT expose per-task-type breakdowns -- it returns flat agent scores
- Agent-registry SKILL.md Section 4 describes recommendation algorithm consuming memory scores
- Task types come from CATALOG.md Section 3 (Task Type Index) -- categories like web-development, api-development, mobile-development, etc.
- See `skills/memory-manager/SKILL.md` for full memory manager specification
- IMPORTANT: The memory-manager is a markdown SPECIFICATION (SKILL.md), NOT executable JavaScript. There is no JS module to import. The "store" and "recall" operations are prose instructions that AI agents follow when reading/writing OUTCOMES.md. Tests for this plan validate the spec document structure, not executable code.
- `tests/memory-manager.test.js` does NOT exist yet -- this plan creates it as a spec validation test file
</context>

<tasks>
### Task 1: Validate task_type in Memory-Manager Store
**Action**: Update memory-manager SKILL.md store operation to enforce task_type population.

1. In Section 3 (Store Operation), add explicit task_type validation:
   - task_type MUST be populated for every outcome record
   - If task_type is not provided by the calling workflow, infer from agent's primary taskType in CATALOG.md
   - Valid task_types: any tag from the agent-registry CATALOG.md Task Type Index (Section 3)
   - If inference fails, use `"general"` as fallback (never leave blank)
2. Add task_type inference rules to the store operation:
   - Primary: use the task_type passed by the execution-tracker
   - Fallback 1: use the first taskType from the executing agent's CATALOG.md entry
   - Fallback 2: `"general"`
3. Document the validation in the SKILL.md with clear examples

**Verify**: Read through updated Section 3 and confirm task_type validation is unambiguous.
**Done when**: Memory-manager store operation guarantees task_type is never empty in OUTCOMES.md records.

### Task 2: Add Archetype Grouping to Memory-Manager Recall
**Action**: Extend memory-manager SKILL.md recall operation to expose task_type groupings alongside flat agent scores.

1. In Section 4 (Recall Operation), add a new return structure `archetypeScores`:
   ```
   archetypeScores: {
     "task_type": {
       agents: ["agent-id-1", "agent-id-2"],
       successRate: 0.85,
       totalOutcomes: 12,
       avgImportance: 3.2,
       topAgent: "agent-id-1"
     }
   }
   ```
2. Archetype grouping aggregates all outcomes by task_type:
   - Groups outcomes by `Task Type` column
   - Per group: computes success_rate, total outcomes, average importance
   - Identifies top-performing agent per task_type (highest success rate with >= 2 outcomes)
   - Applies same recency decay weighting as agent-level scores
3. Document that `archetypeScores` is returned alongside existing `memoryScores` -- callers can use either or both
4. Update agent-registry SKILL.md Section 4 to document that recommendation algorithm may consume `archetypeScores` for archetype-weighted boosts (Plan 06-03 will implement)

**Verify**: Read through updated Section 4 and confirm archetype grouping is clearly specified.
**Done when**: Memory-manager recall operation returns both flat memoryScores and archetype-grouped archetypeScores.
