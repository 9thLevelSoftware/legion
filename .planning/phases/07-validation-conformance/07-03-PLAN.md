<task type="auto">
  <name>Task 1: Create adapter conformance test suite</name>
  <files>tests/adapter-conformance.test.js</files>
  <action>
Create `tests/adapter-conformance.test.js`. This test validates every adapter .md file in `adapters/` (excluding `ADAPTER.md` itself) against the ADAPTER.md specification.

Use `node:test` and `node:assert/strict`. No external dependencies.

Structure the test with `describe` blocks:

**1. YAML Frontmatter — CLI Identity**
For each adapter file, verify:
- `cli`: non-empty string
- `cli_display_name`: non-empty string, contains spaces or proper capitalization
- `version`: non-empty string matching semver-like pattern
- `support_tier`: one of `"certified"`, `"beta"`, `"experimental"`

**2. YAML Frontmatter — Capabilities**
For each adapter file, verify all 5 capability flags exist and are booleans:
- `parallel_execution`: boolean
- `agent_spawning`: boolean
- `structured_messaging`: boolean
- `native_task_tracking`: boolean
- `read_only_agents`: boolean

**3. YAML Frontmatter — Detection**
For each adapter file, verify:
- `detection.primary`: non-empty string
- `detection.secondary`: non-empty string

**4. YAML Frontmatter — Limits & Quirks (new in Phase 7)**
For each adapter file, verify:
- `max_prompt_size`: positive integer (> 0)
- `known_quirks`: array (can be empty, but must be present)
- Each entry in `known_quirks` is a non-empty string matching lowercase-hyphenated format: `/^[a-z0-9-]+$/`

**5. Required Markdown Sections**
For each adapter file, verify the markdown body contains:
- A "Tool Mappings" section (heading with "Tool Mappings" text)
- An "Interaction Protocol" section
- An "Execution Protocol" section

**6. Tool Mappings Completeness**
For each adapter file, check the Tool Mappings table contains entries for the required generic concepts:
- `spawn_agent_personality`
- `spawn_agent_autonomous`
- `spawn_agent_readonly`
- `coordinate_parallel`
- `collect_results`
- `ask_user`
- `model_planning`
- `model_execution`
- `model_check`

**7. Cross-adapter consistency**
- Exactly 9 adapter files exist (excluding ADAPTER.md)
- All adapter `cli` values are unique

For YAML frontmatter parsing, use the regex-based approach from existing tests. Parse the capabilities as a nested object under the `capabilities:` key. For detection, parse as nested under `detection:`.

Example frontmatter parser approach for nested fields:
```javascript
// After finding frontmatter lines between --- delimiters:
// Look for "capabilities:" then indented boolean fields
// Look for "detection:" then indented string fields
// Look for "max_prompt_size:" as top-level integer
// Look for "known_quirks:" then indented "- value" items
```
