**Frontmatter** (follow Phase 29 progressive disclosure pattern):
```yaml
---
name: legion:spec-pipeline

```

**Introduction paragraph:** Brief description of the skill's purpose. Adapted from Auto-Claude's multi-stage spec pipeline. Standalone invocation or offered during `/legion:plan`. Output at `.planning/specs/{NN}-{phase-slug}-spec.md`.

**Section 1: Gather Requirements**
- Input: ROADMAP.md phase details, REQUIREMENTS.md entries, PROJECT.md constraints
- Process: Extract all requirements for the target phase, identify success criteria, list deliverables, note constraints and dependencies
- Output: Structured requirements summary with categorized items (functional, non-functional, constraints)
- The section should document step-by-step extraction like phase-decomposer Section 2

**Section 2: Research Domain**
- Input: Section 1 output + codebase context (if .planning/CODEBASE.md exists)
- Process: Identify knowledge gaps. For each gap: search existing codebase for patterns, read relevant skill/command files, check for prior phase outputs that inform this work. Note unknowns that need user input.
- When to spawn research agents: If the phase involves unfamiliar technology or patterns, spawn 1-2 read-only (Explore) agents to research specific domains. Otherwise, the orchestrator handles research directly.
- Output: Research brief with findings, open questions, and recommended approaches

**Section 3: Write Spec**
- Input: Section 1 requirements + Section 2 research
- Process: Produce a structured specification document at `.planning/specs/{NN}-{phase-slug}-spec.md`
- Spec document structure:
  ```markdown
  # Spec: Phase {N} — {phase_name}

  ## Overview
  {1-paragraph summary of what will be built and why}

  ## Requirements
  {Table: ID | Description | Priority | Acceptance Criteria}

  ## Architecture
  {How the deliverables fit together, key design decisions, file relationships}

  ## Deliverables
  {For each deliverable: file path, purpose, key content, dependencies}

  ## Open Questions
  {Unresolved items from research, items needing user decision}

  ## Complexity Assessment
  {Filled in by Section 5}
  ```
- The spec is a draft at this stage — Section 4 will critique it

**Section 4: Critique Spec**
- Input: Section 3 spec document
- Process: Apply a critical review pass (similar to plan-critique Section 1 pre-mortem):
  a. Are requirements fully addressed or are there gaps?
  b. Are architecture decisions justified or arbitrary?
  c. Are deliverables concrete enough for an agent to implement without guessing?
  d. Are there implicit assumptions that should be explicit?
  e. Are open questions actually blockers or can they be deferred?
- Agent selection: Optionally spawn a skeptical agent (testing-qa-verification-specialist or product-sprint-prioritizer) using the read-only Explore subagent type to critique the spec. For simple phases, the orchestrator can self-critique.
- Output: Critique findings with specific revision recommendations. Update the spec document with revisions.

**Section 5: Assess Complexity**
- Input: Revised spec from Section 4
- Process: Estimate phase complexity to inform plan structure:
  - Count deliverables and categorize (new file, modification, config change)
  - Estimate dependency depth (how many waves likely needed)
  - Rate overall complexity: Simple (1 plan likely), Medium (2 plans), Complex (3+ plans)
  - Recommend whether competing architecture proposals (PLN-01) would add value for this phase
- Output: Complexity rating and recommendation added to spec document's "Complexity Assessment" section

**Section 6: Integration Points**
Document how spec-pipeline connects with other workflows:
- `/legion:plan` — offers spec pipeline as optional step; spec output feeds into phase decomposition
- `/legion:build` — agents can reference spec document for implementation guidance
- `plan-critique` — spec critique (Section 4) reuses the same agent selection and read-only spawning pattern
- `workflow-common` — spec documents stored at `.planning/specs/{NN}-{phase-slug}-spec.md`

**Section 7: Standalone Invocation**
Document how to invoke the spec pipeline outside of `/legion:plan`:
- User runs the pipeline on a specific phase number
- All 5 stages execute sequentially
- Output: spec document at `.planning/specs/` ready for subsequent planning
- Can be re-run to update a spec after requirements change

**References table:** Map each section to its inspiration source (Auto-Claude for pipeline stages, plan-critique for critique pattern, phase-decomposer for extraction patterns).
