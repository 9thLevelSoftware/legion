This is an **optional** field (unlike `verification_commands` which is mandatory). When present, it declares files that must be accessed by only one agent at a time during wave execution, even when parallel execution is enabled.

Add a row to the "Template Field Guidance" table:

| Field | How to Fill |
|-------|-------------|
| `sequential_files` | Optional list of file paths that require single-agent sequential access within the wave. Use when multiple plans in the same wave read/write a shared file that isn't in `files_modified` (e.g., a shared config file read by all plans, a lock file, or a generated index file that multiple plans append to). Files in `sequential_files` must NOT also appear in `files_modified` — if a plan modifies a file, declare it in `files_modified` and use `files_forbidden` in other plans to prevent conflicts. Empty array `[]` or omission means no sequential constraints. |

Add a subsection after "Writing expected_artifacts Declarations" titled "Writing sequential_files Declarations":

```markdown
### Writing sequential_files Declarations

Use `sequential_files` when a wave has shared resources that multiple plans need to access but not modify exclusively. This is rare — most parallelism conflicts are resolved by wave separation.

**When to use sequential_files:**
- A shared index or manifest file that multiple plans append to
- A generated file that one plan writes and others read within the same wave
- A lock file or semaphore used for coordination

**When NOT to use sequential_files:**
- Files that a plan modifies exclusively → use `files_modified` instead
- Files that other plans must never touch → use `files_forbidden` instead
- Dependencies between plans → use `depends_on` to put them in different waves

**Example:**
```yaml
# Plan A and Plan B are both in Wave 1.
# Both need to append to a shared CHANGELOG.md.
sequential_files:
  - CHANGELOG.md
```

**Wave-executor behavior:** When `sequential_files` is present, the wave-executor orders dispatch so that plans touching the same sequential file execute one at a time for that file's access, even when `adapter.parallel_execution` is true. Plans with no sequential_files overlap still execute in parallel.
```
