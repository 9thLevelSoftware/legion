## Task 1: Define metadata field schema

Define the schema for the 4 new agent frontmatter fields. Each field is a YAML array.

**Field definitions:**

1. **`languages`** -- Programming/markup languages the agent works with
   - Examples: `[javascript, typescript, python, php, swift, html, css, sql]`
   - For non-technical agents (marketing, design, PM): use `[markdown]` or `[markdown, yaml]`
   - Must reflect what the agent's personality text actually describes

2. **`frameworks`** -- Frameworks, libraries, and tools the agent specializes in
   - Examples: `[react, vue, angular, laravel, django, kubernetes, figma]`
   - For non-technical agents: use domain tools like `[google-analytics, semrush, figma, jira]`
   - Must be specific to the agent's described expertise

3. **`artifact_types`** -- Types of outputs the agent produces
   - Examples: `[code, tests, api-specs, documentation, design-systems, reports, campaigns]`
   - Drawn from the agent's "Technical Deliverables" section

4. **`review_strengths`** -- What this agent is best at reviewing/critiquing
   - Examples: `[code-quality, performance, security, accessibility, brand-consistency, test-coverage]`
   - Drawn from the agent's expertise and anti-patterns sections

**Schema rules:**
- All 4 fields are required (no empty arrays -- minimum 1 value each)
- Values are lowercase, hyphenated (no spaces, no camelCase)
- Values should be specific to the agent, not generic across all agents
- Maximum 8 values per field (keeps metadata focused)

**Action:** Document this schema as a comment block at the top of the test file for reference during enrichment.

**Verify:** Schema is internally consistent and covers all 9 divisions.

**Done when:** Schema definition is documented and ready for test implementation.

## Task 2: Update agent contract tests

Extend `tests/agent-contract.test.js` to validate the 4 new frontmatter fields.

**YAML Parsing Strategy:** Use regex-based frontmatter extraction -- no new npm dependency needed. Split file content on `---` fences, take the block between first and second fence, and parse the 4 known array fields using a regex like `/^(languages|frameworks|artifact_types|review_strengths):\s*\[(.+)\]/` to extract YAML flow sequences. This is safe because the frontmatter schema is constrained (flat arrays of hyphenated strings only).

**Action:**
1. Add a `parseFrontmatter(text)` helper function using the regex approach above
2. Add TWO new test cases:
   a. `agent contract: metadata fields valid when present` (format validation)
      - For each of the 53 agent files:
        - Parse YAML frontmatter
        - If metadata fields exist, validate:
          - Each field is a non-empty array of strings
          - All values are lowercase and match `/^[a-z0-9-]+$/`
          - No field has more than 8 values
        - If metadata fields are absent, SKIP (do not fail) -- this tolerates partial enrichment during Wave 2 parallel execution
   b. `agent contract: all 53 agents have metadata (completeness gate)` (completion check)
      - Assert that ALL 53 agents have all 4 metadata fields present
      - This test is expected to FAIL until both Plans 05-02 and 05-03 complete
      - After Phase 5 is fully done, this becomes the permanent completeness gate
3. Keep existing tests unchanged -- add new test cases only

**Why two tests:** Plans 05-02 and 05-03 run in parallel (both Wave 2). Each plan can only enrich its own subset. The format validation test passes for partially-enriched state, so each plan can verify its agents pass format checks. The completeness gate only passes when all 53 are done.

**Verify:** `node --test tests/agent-contract.test.js` -- format validation passes for 3 exemplar agents, completeness gate fails (expected). Existing tests still pass.

**Done when:** Both new test cases are implemented and the 3 exemplar agents pass format validation.
