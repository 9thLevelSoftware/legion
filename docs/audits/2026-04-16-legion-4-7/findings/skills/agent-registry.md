**Issue:** LEGION-47-085 INVERSE CHECK (per task brief). (b) L174-249 inlines 20+ agent IDs in team-assembly tables AS STRINGS with no reference to the CATALOG, frontmatter, or CLAUDE.md index. If any agent is renamed, deprecated, or added, these inline lists silently drift. (c) The inline lists are themselves the SEVENTH occurrence of non-existent-agent-ID class risk if any of the 20+ IDs is stale. Cross-checking: I verified L183-249 IDs against CLAUDE.md Division Index — all IDs appear in known divisions, but `engineering-laravel-specialist` at L238 is a narrower pattern, `xr-immersive-developer` at L248 is ungrouped in the index. Peer LEGION-47-085, 084, 187, 199, 225.

(c) Add CI validation: scripts/audit/validate-agent-registry.sh compares (1) `ls agents/*.md` vs (2) CATALOG.md rows vs (3) CLAUDE.md Division Index counts — fail build on drift. (d) Cross-reference LEGION-47-085 (primary dual-ground-truth finding for intent-teams.yaml — same pattern applies here); propose consolidated `scripts/audit/validate-agent-id-refs.sh` walking every .md file in repo and flagging any agent-id string not in agents/*.md. Immediate hard blocker for S11.

## LEGION-47-235 — P2, Dead Metadata (confirmed)

**Lines 128-149**

> ### Score Export
> After scoring, produce a structured score breakdown for each recommended agent:
> ```
> score_export:
>   task_type_detected: "{extracted task type from TASK_TYPE_MAP}"
>   candidates:
>     - agent_id: "{agent-id}"
>       semantic_score: {numeric from Layer 1}
>       heuristic_score: {numeric from Layer 2}
>       metadata_score: {numeric from Layer 3}
>       memory_boost: {numeric from OUTCOMES.md or 0}
>       archetype_boost: {numeric from archetype formula or 0}
>       total_score: {semantic + heuristic + metadata + memory + archetype}
>       confidence: "{HIGH|MEDIUM|LOW}"
>     - agent_id: "{agent-id-2}"
>       ...
>   adapter: "{adapter name from current CLI}"
>   model_tier: "{planning|execution|check}"
>   recommendation_source: "{semantic|heuristic|memory|archetype|override}"
```

**Issue:** LEGION-47-110 origin point (per task brief). L147 `model_tier: "{planning|execution|check}"` is the ORIGINAL declaration — emitted at recommendation time per L152 "Produced by agent-registry during recommendation". Dead-metadata chain: (a) No consumer documented. Agent-registry does NOT produce model_tier — it would need to be passed IN by the caller (wave-executor, /legion:plan, /legion:build). But none of the three specs said. (b) L153 says wave-executor consumes score_export — checked in S06, wave-executor SUMMARY.md export is OBS-01 and does NOT actually write model_tier per audited structure. (c) The tier values {planning|execution|check} correspond to cost-profile tier names from CLAUDE.md Conventions L164 "Cost profile: Planning/execution/check tiers mapped to CLI-specific model names via adapter" — but the adapter mapping is static and adapter-specified, not per-recommendation. Emitting model_tier per-candidate implies different agents get different tiers per call — NOT the case; tier is determined by WHAT OPERATION is being run, not BY WHICH agent. The field is semantically invalid at the candidate level. (d) `recommendation_source: "{semantic|heuristic|memory|archetype|override}"` does not include "mandatory" despite L167 "Mandatory role additions should be reflected in the score_export with `recommendation_source: \"mandatory\"`" — closed set missing one documented value.

**Remediation sketch:** (a) Decide fate of model_tier. Option 1: Remove from per-candidate score_export — it's a RUN-level attribute belonging at the top of score_export not per-candidate. Move to `run_metadata: { model_tier: ..., adapter: ... }`. Option 2: Delete entirely — if no consumer reads it, drop the field (peer LEGION-47-110). (b) L148 closed set: add `mandatory` to the allowed values to match L167 declaration. (c) Add "Consumer Contract" subsection: list which fields each consumer (wave-executor, planner, summary exporter) reads — unread fields flagged for deletion. (d) Cross-reference LEGION-47-110 (declared dead metadata); this finding answers "where is it originally declared" — here, without consumer spec.

**Remediation cluster:** `dispatch-specification`

## LEGION-47-238 — P2, Maximalist Language (suspected)

**Lines 160, 162-166, 167**

> - Never exceed 4 agents for one discrete task; split larger efforts.
> ...
> - **Execution teams** (code-writing/deployment) MUST include at least one Testing-division agent.
> - **Cross-division teams** MUST include a coordinator: `project-manager-senior`, `project-management-project-shepherd`, or `agents-orchestrator`.
> - **User-facing change teams** SHOULD include a Design-division reviewer.
> ...
> Mandatory role additions should be reflected in the score_export with `recommendation_source: "mandatory"`.

**Issue:** Absolute constraints without escape valves, CAT-7 peer LEGION-47-125, 140, 155. (a) L160 "Never exceed 4 agents" is a hard cap but real-world cross-domain phases may need 5+ (e.g., launches) — the next sentence "split larger efforts" assumes split is always possible, which conflicts with atomic tasks (e.g., cross-platform compatibility review). (b) L162 MUST include Testing-division for execution — a bug-fix sub-plan in a stable module may not need a test agent if full suite passes unchanged. Current rule forces artificial Testing inclusion. (c) L164 MUST include coordinator for cross-division — 2-agent cross-division teams (e.g., 1 engineering + 1 design for a UI polish) must now include a third coordinator, violating L157 "2 agents for single-domain" since cross-division is now required to be 3+. (d) L167 "Mandatory role additions should be reflected" — "should" is not enforced; passive voice. No validator checks that recommendation_source:"mandatory" is actually attached.

**Remediation sketch:** (a) L160 soften to "Prefer ≤4 agents; if >4 is required by task spec, emit an `<escalation severity=warning type=scope>` asking the user to confirm the larger team OR split." (b) L162 add exception: "MUST include Testing-division UNLESS the plan.skip_testing flag is set with documented rationale in plan frontmatter. Rationale acceptable: no code change (docs only), pure config change with no runtime impact." (c) L164 scope to 3+ agents: "Cross-division teams of 3+ agents MUST include a coordinator. 2-agent cross-division teams MAY omit a coordinator." (d) L167 replace "should" with enforceable contract: "MUST emit recommendation_source='mandatory' on the added agent's score_export entry. Validator: grep 'recommendation_source: "mandatory"' count == mandatory-role-addition count." Cross-reference LEGION-47-125, 140.

**Remediation cluster:** `persona-calibration`
**Effort estimate:** small
