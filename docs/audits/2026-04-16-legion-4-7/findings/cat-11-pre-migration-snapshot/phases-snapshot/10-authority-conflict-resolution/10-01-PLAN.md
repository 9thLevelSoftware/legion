## Task 1: Define Escalation Protocol Specification

**Action**: Create `.planning/config/escalation-protocol.yaml` defining the structured escalation format, severity levels, routing rules, and control mode behaviors.

1. Define the escalation block format that agents must use when encountering out-of-scope decisions:
   ```yaml
   escalation_format:
     block_tag: "escalation"  # agents wrap escalations in <escalation> blocks
     required_fields:
       - severity    # info | warning | blocker
       - type        # architecture | dependency | scope | schema | api | deletion | infrastructure | quality
       - decision    # what decision is needed (one sentence)
       - context     # why the agent encountered this (2-3 sentences)
       - alternatives # what the agent would do if authorized (optional)
     optional_fields:
       - affected_files  # files that would be modified
       - related_domain  # authority matrix domain this falls under
   ```

2. Define severity levels with routing behavior:
   - `info`: Log to SUMMARY.md, continue execution. Human reviews post-build.
   - `warning`: Log to SUMMARY.md, continue execution. Highlighted in build output.
   - `blocker`: Log to SUMMARY.md, halt current task. Agent continues other in-scope tasks. Human must resolve before blocked task can proceed.

3. Define escalation type taxonomy (aligned with CLAUDE.md "Human Approval Required" table):
   - `architecture`: New patterns, abstractions, structural changes
   - `dependency`: Unplanned package/library additions
   - `scope`: Modifications outside task's `files_modified` list
   - `schema`: Database schema changes
   - `api`: Endpoint or contract changes
   - `deletion`: Removing existing functionality
   - `infrastructure`: CI/CD or deployment changes
   - `quality`: Overriding review findings or skipping gates

4. Define control mode behavior for escalations:
   - `autonomous`: Escalations logged but agent proceeds without halting (info-level treatment for all)
   - `guarded` (default): Full escalation protocol — severity-based routing as defined above
   - `advisory`: All decisions logged as escalations (read-only mode, everything is informational)
   - `surgical`: Strict escalation — any out-of-scope file access is automatic `blocker`

5. Define escalation resolution tracking:
   - Resolution statuses: `pending` | `approved` | `rejected` | `deferred`
   - Resolution is recorded in SUMMARY.md alongside the original escalation
   - No separate registry file — escalations live in the plan's SUMMARY.md to keep state human-readable and co-located with execution context

**Verify**: `cat .planning/config/escalation-protocol.yaml | head -30` shows format definition
**Done when**: Escalation protocol YAML defines format, severities, types, control mode behaviors, and resolution tracking

## Task 2: Integrate Escalation Detection into Wave Executor

**Action**: Update `skills/wave-executor/SKILL.md` to detect escalation blocks in agent output and route them according to the protocol.

1. Add a new section (after existing result collection, before SUMMARY.md writing) — "Escalation Detection & Routing":
   - After each agent completes a task, scan output for `<escalation>` blocks
   - Parse each escalation block and extract severity, type, decision, context
   - Validate required fields are present; if malformed, treat as `warning` severity with note about format

2. Add escalation routing logic:
   - For `info` escalations: append to SUMMARY.md "Escalations" section, continue
   - For `warning` escalations: append to SUMMARY.md, display to orchestrator with highlight, continue
   - For `blocker` escalations:
     a. Append to SUMMARY.md with `status: pending`
     b. Display to user via `adapter.ask_user`:
        ```
        "Agent {name} flagged a blocker escalation:"
        {decision}
        {context}
        → Approve: Allow the agent to proceed with the proposed action
          Reject: Agent skips this action and continues other tasks
          Defer: Mark for later resolution, agent continues other tasks
        ```
     c. Record resolution in SUMMARY.md

3. Add control mode check before escalation routing:
   - Read current `control_mode` from settings
   - Apply mode-specific behavior from escalation-protocol.yaml
   - In `autonomous` mode: log all escalations as info, never halt
   - In `surgical` mode: auto-generate `blocker` escalation for any file not in `files_modified`

4. Add "Escalations" section to the SUMMARY.md template:
   ```markdown
   ## Escalations

   | Severity | Type | Decision | Status |
   |----------|------|----------|--------|
   | blocker | scope | Need approval to modify utils.js | approved |
   | warning | dependency | Considering adding lodash for deep merge | logged |
   ```
